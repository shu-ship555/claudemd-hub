## 8. ブランチ & コミット規約 (Git Conventions)

<!-- エージェントがGit操作を行う際のルール -->

### ブランチ命名
```
feature/[issue番号]-[簡潔な説明]   例: feature/42-add-user-auth
fix/[issue番号]-[簡潔な説明]       例: fix/88-null-pointer-login
chore/[説明]                        例: chore/update-dependencies
```

### コミットメッセージ (Conventional Commits)
```
<type>(<scope>): <summary>

[オプション: 詳細説明]

[オプション: フッター / BREAKING CHANGE]
```

**type一覧:**
| type | 用途 |
|------|------|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `refactor` | 機能変更を伴わないリファクタリング |
| `test` | テストの追加・修正 |
| `chore` | ビルド・依存関係・ツール設定の変更 |

### PRのルール
- 1 PR = 1つの目的（機能追加とリファクタは分ける）
- PRテンプレートがある場合は必ず埋める
- レビュー前にセルフレビューを行う

## 9. エージェントへの行動指示 (Agent Behavior Rules)

<!-- AIエージェントが自律的に動く際の制約・優先順位を明示する -->

### 必ずすること (MUST)
- 変更前に既存のテストが通ることを確認する
- 新しいファイルを作成する前に既存ファイルを確認する
- 不明な仕様は推測せず、コメント or ドキュメントを確認する
- 破壊的変更（APIの削除・DB スキーマ変更）は実行前に確認を求める

### してはいけないこと (MUST NOT)
- `.env` ファイルや秘密情報をコミットする
- `main` / `master` ブランチへ直接プッシュする
- テストを削除してカバレッジを下げる
- 本番データベースに対してDDLを直接実行する
- [プロジェクト固有の禁止事項を追記]

### 推奨される行動 (SHOULD)
- 大きな変更は小さなステップに分割して実行する
- 既存コードのパターンに従い、スタイルを統一する
- 変更範囲が広い場合は計画を提示してから実行する

### 判断に迷ったとき
1. このAGENT.mdを再確認する
2. 既存コードの慣例を参照する
3. 関連するテストから仕様を読み取る
4. それでも不明な場合は**実行せず、質問を提示する**
